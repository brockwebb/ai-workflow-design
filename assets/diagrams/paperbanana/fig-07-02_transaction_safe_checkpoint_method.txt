# fig-07-02: Transaction-Safe Checkpoint Write Pattern

## Diagram Type
Flow diagram showing a sequential process with failure branches. NOT an architecture diagram. Linear happy path left-to-right. Crash failure branches drop downward from each step. Engineering pattern diagram.

## Layout
Top section: left-to-right happy path with 4 steps.
Below each step: a dashed downward arrow to a crash outcome box.
Bottom annotation row: summary of the safety guarantee.

## Elements

### Happy Path (top row, left to right)

1. Box: "New checkpoint data ready"
   Arrow right →

2. Box: "Write to checkpoint.tmp"
   Small text: "(temporary file)"
   Arrow right →

3. Box: "Atomic rename: checkpoint.tmp → checkpoint.json"
   Small text: "(single OS operation)"
   Arrow right →

4. Box: "checkpoint.json updated"
   Small text: "Success"
   Style: slightly distinct fill to indicate completion

### Crash Branch 1 (below step 2)
Dashed arrow downward from "Write to checkpoint.tmp" →
Crash box: "Crash during write"
Small text: "checkpoint.tmp is partial or missing. checkpoint.json is the previous valid checkpoint. Resume is safe from last checkpoint."
Style: amber or light orange fill (#FFF3CD) to indicate failure path, not destructive

### Crash Branch 2 (below step 3)
Dashed arrow downward from "Atomic rename" →
Crash box: "Crash during rename"
Small text: "Rename is atomic on POSIX systems. checkpoint.json is either the old version or the new version. Never a partial file."
Style: same amber fill as Crash Branch 1

### Bottom Annotation
Horizontal rule or separator.
Text: "Never write directly to checkpoint.json — a crash during a direct write leaves a partial, unreadable checkpoint."
Style: italic or caption-weight text, medium gray (#666666)

## Visual Style
- Engineering flow diagram — rectangular boxes, clean arrows
- Happy path boxes: white (#FFFFFF) fill, dark gray (#333333) outline
- Completion box (step 4): light green (#F0FFF0) fill to indicate success
- Crash outcome boxes: light amber (#FFF3CD) fill — failure paths, not errors
- Happy path arrows: solid dark gray (#333333)
- Crash branch arrows: dashed, medium gray (#888888)
- No decorative elements, no icons
- Sans-serif font throughout
- Generous whitespace between rows

## Dimensions
- Landscape orientation, roughly 3:1 aspect ratio (wide and relatively short)
- Full page width

## What This Is NOT
- Not a sequence diagram with actors
- Not a state machine diagram
- The crash boxes show outcomes, not error handlers — the diagram explains the safety guarantee, not the recovery code
- Do not show file system internals or OS-level details beyond "atomic rename"

## Caption
"Transaction-safe checkpoint write pattern. Writing to a temp file and atomically renaming prevents checkpoint corruption on crash. A crash during the write phase leaves the old checkpoint intact. A crash during the rename (atomic on POSIX systems) leaves either the old or new checkpoint intact, never a partial file."
