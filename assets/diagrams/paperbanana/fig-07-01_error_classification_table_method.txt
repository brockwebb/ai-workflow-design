# fig-07-01: Error Classification Table

## Diagram Type
Four-column reference table formatted as a figure. NOT a flow diagram. Three data rows (plus header), one per error type. Operational quick-reference for practitioners debugging pipeline failures. Book-quality, clean.

## Layout
Table with header row + 3 data rows. Four columns.

## Table Structure

### Header Row
| Error Type | Examples | Pipeline Response | Priority |

### Row 1
| Transient | HTTP 429 (rate limit), network timeout, HTTP 503 | Exponential backoff with jitter; retry up to max count | Retry first; escalate if max retries exceeded |

### Row 2
| Permanent | Invalid API key, deprecated model, removed endpoint, malformed request | Log with full context; stop affected batch; alert operator | Stop immediately; do not retry |

### Row 3
| Data | Malformed input, unparseable model response, schema mismatch | Log failure with input; skip record; continue processing | Review in batch after run; quality signal, not infrastructure signal |

## Visual Style
- Clean book-quality table, not a web UI table
- Header row: dark gray background (#333333), white text
- Error Type column (left): light gray background (#F0F0F0), dark text — serves as row label
- Content cells: white background (#FFFFFF)
- Alternating row shading: Row 1 white, Row 2 very light gray (#FAFAFA), Row 3 white
- Cell borders: thin light gray (#CCCCCC)
- Font: same sans-serif as other figures
- Moderate cell padding — readable, not bloated
- No icons, no color coding on error type names

## Column Widths
- Error Type: narrow (15% of width) — single-word label
- Examples: moderate (25%)
- Pipeline Response: moderate (35%) — most detailed column
- Priority: moderate (25%)

## Dimensions
- Landscape orientation — four columns need width
- Full column width in book layout

## What This Is NOT
- Not a flowchart showing error handling sequence
- Not a decision tree
- No vendor names, no product names
- Priority column is not ranked 1-2-3; it describes response priority within the error type

## Caption
"Error classification and pipeline response. The key design decision is that the pipeline must classify each error type and respond differently. Retrying permanent errors wastes resources. Stopping on data errors prevents completion. Ignoring transient errors produces incomplete results. Error type determines response."
