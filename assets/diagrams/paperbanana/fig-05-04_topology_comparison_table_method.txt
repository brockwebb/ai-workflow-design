# fig-05-04: Topology Comparison Table

## Diagram Type
Structured comparison table. NOT a flow diagram. Clean, readable, book-quality table layout. This is the highest-ROI visual in Chapter 5.

## Layout
Three-column table (one column per topology) with a left-side row-label column. Seven rows of comparison criteria. Header row at top with topology names. Table is wider than tall (landscape orientation).

## Table Structure

### Header Row
| (blank) | Parallel Consensus | Generator-Critic Loop | LLM-as-Judge |

### Row 1: Task Type
| Task type | Classification, coding, categorization with discrete outputs | Complex generation or extraction against a formal schema | Quality gating, model selection, multi-dimensional evaluation |

### Row 2: Output Relationship
| Output relationship | N models, same task, independent | Generator produces; critic evaluates (different tasks in sequence) | Judge evaluates a single output against defined criteria |

### Row 3: Evaluation Method
| Evaluation method | Inter-rater reliability (kappa, alpha) | Schema conformance + trajectory monitoring | Rubric scores per dimension |

### Row 4: Cost Profile
| Cost profile | 2-3x inference cost per item | Variable: base cost × iterations; critic adds per-loop overhead | Base cost + judge inference; one pass unless human escalation |

### Row 5: Primary Failure Mode
| Primary failure mode | Correlated errors when models share training; low disagreement masks systematic bias | Oscillation (iteration N+1 worse than N); reward hacking if generator games critic | Rubric gaming; judge drift over time; preference leakage from related models |

### Row 6: Use When
| Use when | Disagreement is itself the signal; volume justifies 2-3x cost; inter-rater framework applies | Partial correctness is typical; schema conformance is testable; retry cost beats human review cost | Quality gate before production; scalable alternative to human review; prompt optimization experiments |

### Row 7: Do NOT Use When
| Do not use when | Models are correlated on your specific domain (diversity is not available); volume is too low to amortize cost | No observable trajectory signal; critic is more expensive than human review; task lacks schema | Domain knowledge exceeds judge capability; rubric cannot be defined precisely; no calibration process in place |

## Visual Style
- Clean book-quality table, not a web table
- Header row: dark gray background (#333333), white text
- Row label column (leftmost): light gray background (#F0F0F0), dark text, left-aligned
- Content cells: white background, dark text
- Alternating row shading for readability: even rows white (#FFFFFF), odd rows very light gray (#FAFAFA)
- Cell borders: thin, light gray (#CCCCCC)
- Text: concise, sentence fragments acceptable — this is a reference table, not prose
- Font: same sans-serif as other figures (Inter, Helvetica)
- No icons, no color coding beyond the header/label shading
- Moderate padding in cells — enough for readability, not so much the table bloats

## Dimensions
- Landscape orientation
- Full page width
- Aim for rows that are readable without wrapping too severely — compress text where needed

## What This Is NOT
- Not a flow diagram
- Not a comparison of vendors or specific models
- Do not add "winner" indicators, checkmarks, or star ratings
- Row labels are descriptive categories, not numeric rankings
- The "Do not use when" row is as important as the "Use when" row

## Caption
"Topology comparison. Three multi-model topologies compared across task type, evaluation method, cost profile, primary failure mode, and use conditions. The right topology depends on task structure, cost constraints, and what signal you are optimizing for."
